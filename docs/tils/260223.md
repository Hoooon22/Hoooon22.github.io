## 리눅스 시스템의 이해

### set

셸에서 선언된 셸 변수 전부를 확인할 때 사용하는 명령어

### PS1, PS2, PS3, PS4

프롬프트의 모양을 지정하는 환경 변수

PS1: Primary Prompt String
PS2: Secondary Prompt String
PS3: Third Prompt String
PS4: Fourth Prompt String

### 위치 매개변수

만약 파일명이 test.sh이고, ./test.sh lin joon이라고 실행했다면, 출력은 다음과 같다.

```
./test.sh
lin
```

$0: 실행된 스크립트 파일의 이름을 나타냄. (출력: ./test.sh)

$1: 명령행에서 전달된 첫 번째 인자. (출력: lin)

$2: (만약 코드에 있었다면) 두 번째 인자인 **joon**이 출력.

### True False

리눅스에서 0은 성공(참)을 의미하고, 1은 실패(거짓)를 의미한다.

### PID 0, 1

| 구분            | PID 0                            | PID 1                                 |
| :-------------- | :------------------------------- | :------------------------------------ |
| **이름**        | Swapper / Idle                   | init (과거) / systemd (현재)          |
| **생성 주체**   | 커널이 직접 생성                 | PID 0번에 의해 생성됨                 |
| **역할**        | 프로세스 스케줄링, 시스템 초기화 | 모든 사용자 프로세스의 부모           |
| **사용자 공간** | 커널 공간에서만 존재             | 사용자 공간(User Space)의 첫 프로세스 |

### 시그널 번호

| 번호 | 시그널 이름 | 설명 및 용도                                                                                                        |
| :--- | :---------- | :------------------------------------------------------------------------------------------------------------------ |
| 1    | **SIGHUP**  | Hangup. 터미널과 연결이 끊겼을 때 발생. 보통 데몬 프로세스(서비스)의 **설정 파일을 재설정(Reload)**할 때 사용.      |
| 2    | **SIGINT**  | Interrupt. 키보드로부터의 종료 신호. **Ctrl + C**를 누를 때 발생.                                                   |
| 3    | **SIGQUIT** | Quit. 키보드로부터의 종료 신호이지만, 종료 시 **코어 덤프(Core Dump)**를 생성. Ctrl + \ 누를 때 발생.               |
| 9    | **SIGKILL** | Kill. 프로세스를 강제 종료. 프로세스가 가로채거나 무시할 수 없는 가장 강력한 신호.                                  |
| 15   | **SIGTERM** | Terminate. 프로세스에게 정상적인 종료를 요청. kill 명령어의 기본 시그널.                                            |
| 18   | **SIGCONT** | Continue. SIGSTOP 등에 의해 정지된 프로세스를 다시 재개.                                                            |
| 19   | **SIGSTOP** | Stop. 프로세스를 즉시 정지. SIGKILL처럼 무시할 수 없음.                                                             |
| 20   | **SIGTSTP** | Terminal Stop. 키보드로부터의 정지 신호. **Ctrl + Z**를 누를 때 발생. 프로세스를 배경(Background)으로 보낼 때 사용. |

### systemd 유틸리티 (ctl 명령어)

systemd 시스템 환경에서 전반적인 제어(control)를 담당하는 명령어들이다.

| 명령어          | 설명                                                                                                     |
| --------------- | -------------------------------------------------------------------------------------------------------- |
| **systemctl**   | 시스템 서비스(데몬) 시작/중지 및 시스템 상태(런레벨 등)를 총괄적으로 관리하는 가장 핵심적인 명령어       |
| **machinectl**  | systemd 내의 컨테이너 및 가상 머신(VM) 상태를 등록, 확인 및 제어하는 명령어                              |
| **hostnamectl** | 시스템의 호스트네임(컴퓨터 이름)을 조회하거나 영구적으로 설정(변경)하는 명령어                           |
| **journalctl**  | `systemd-journald` 데몬이 수집한 시스템 부팅 및 서비스 전체 로그(저널 데이터)를 조회하고 검색하는 명령어 |

### GRUB 부트 메뉴 단축키

부팅 시 나타나는 GRUB 화면에서 사용하는 주요 단축키이다. (단일 사용자 모드 부팅 등으로 진입할 때 필수적으로 알아야 하는 키들이다.)

- **`e` (Edit)**: 선택한 부팅 항목의 설정을 임시로 **편집(수정)**한다. 런레벨 변경(1번 등)이나 비밀번호 분실 시 복구를 위해 파라미터를 추가할 때 가장 많이 사용한다.
- **`c` (Command-line)**: GRUB의 **명령행 프롬프트** 모드로 이동하여 커맨드를 직접 입력할 수 있게 한다.
- **`a` (Append)**: 커널 매개변수 끝에 내용을 **추가**한다. (주로 기존 GRUB Legacy에서 많이 썼고, 최근에는 'e'로 통합하여 편집하는 방식을 선호한다.)
- **`b` (Boot)**: 편집(e) 모드에서 설정을 마친 후 시스템을 **부팅**한다. (최신 GRUB2에서는 `Ctrl + x` 또는 `F10`을 부팅 키로 주로 사용한다.)

## 네트워크의 이해

### DQDB

DQDB(Distributed Queue Dual Bus)는 주로 **도시권 통신망(MAN, Metropolitan Area Network)** 구축에 사용되는 네트워크 구조이자 IEEE 802.6 표준 규격이다.

- **두 개의 단방향 버스**: 양방향 전송을 위해 2개의 단방향 광섬유 라인(Dual Bus)으로 구성되어 있다.
- **충돌 없는 슬롯 통신**: 회선을 작은 크기(53바이트 셀)의 '빈 슬롯'으로 쪼개서 전송 데이터를 싣는다. 토큰 패싱 방식과 달리 빈 슬롯이 오면 바로 채워서 보내기 때문에 노드 간 충돌이 발생하지 않으며 대기 지연이 적다.
- **다양한 서비스 통신망 지원**: 텍스트, 음성, 화상(비디오) 등 대용량 데이터 전송에 적합하며 B-ISDN(초고속 정보통신망)과 호환성이 높다.

### 회선 교환 방식 (Circuit Switching)

데이터를 전송하기 전에 송신측과 수신측 사이에 하나의 전용 물리적 통신 경로(회선)를 미리 연결해 놓은 뒤 통신하는 방식이다. **"옛날 전화기(전화망)"**를 떠올리면 가장 이해하기 쉽다.

- **독점적인 경로 연결**: 연결이 유지되는 동안에는 이 회선을 양쪽 호스트가 독점하여 사용하기 때문에 다른 사용자는 이 회선에 끼어들 수 없다.
- **실시간 통신 유리**: 전송 지연이 거의 없어 음성 통화나 실시간 스트리밍 등 타이밍이 중요한 통신에 매우 적합하다.
- **낮은 효율성**: 데이터를 보내지 않고 잠시 쉬는 시간(침묵)에도 회선을 꽉 잡고 있으므로, 네트워크 전체의 효율성 측면에서는 대역폭 낭비가 발생한다.
- **연결 설정 지연**: 상대방 시스템과 통신을 시작하기 전, 초기에 물리적인 회선을 연결(Setup)하는 데 일정 시간이 소요된다.

(참고: 회선 교환 방식의 반대되는 개념으로는 인터넷망의 기본인 패킷을 잘게 쪼개 보내는 **패킷 교환 방식(Packet Switching)**이 있다.)

### 패킷 교환 방식 (Packet Switching)

데이터를 **패킷(Packet)**이라는 작은 단위로 쪼개어 전송하는 방식으로, 현재 **인터넷 통신의 기본 방식**이다.

- **경로 비독점 (우회 가능)**: 고정된 회선 없이 패킷마다 가장 빠르고 빈 경로를 찾아 이동하므로, 회선 장애 시 우회 전송이 가능하다.
- **높은 회선 효율성**: 여러 사용자가 회선을 공유하므로, 네트워크 대역폭을 매우 효율적으로 사용할 수 있다.
- **지연 및 순서 뒤섞임 발생**: 각 패킷이 도착하는 시간과 순서가 다를 수 있어, 수신 측에서 이를 재조립해야 한다. (실시간 통신에는 다소 불리할 수 있음)
- **방식 구분**: 연결 지향형인 **가상 회선 방식(TCP 등)**과 비연결형인 **데이터그램 방식(UDP 등)**으로 나뉜다.

### 네트워크 접속 장비 특성

| 장비명                   | OSI 계층           | 특징 및 역할                                                                                                                                                    |
| ------------------------ | ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Repeater (리피터)**    | 1계층 (물리)       | 신호 증폭기. 감쇠된 전기 신호를 단순히 재생 및 파워업시켜 전송 거리를 연장해 준다.                                                                              |
| **Hub (허브)**           | 1계층 (물리)       | 여러 PC를 연결해주는 멀티포트 장치. 들어온 데이터를 모든 포트로 뿌린다(Broadcasting). 그래서 충돌(Collision)이 자주 발생한다.                                   |
| **Bridge (브리지)**      | 2계층 (데이터링크) | 허브와 비슷하지만, **MAC 주소**를 구별하여 필요한 곳으로만 데이터를 보낸다. 충돌 영역(Collision Domain)을 나눠주는 역할을 한다.                                 |
| **Router (라우터)**      | 3계층 (네트워크)   | 서로 다른 형태의 통신망(네트워크)을 연결해 주고, 목적지까지 가는 가장 빠르고 **최적의 경로(Routing)**를 찾아 트래픽을 안내해 주는 핵심 장비다. (IP 주소 기반)   |
| **Gateway (게이트웨이)** | 4~7계층 (전 계층)  | 가장 많은 계층(OSI 전체 데이터 전송 단계)을 지원하는 장비. 프로토콜, 구조(아키텍처)가 완전히 다른 두 개의 네트워크망을 이어주는 '출입문'이자 거대한 통역사이다. |

### TCP/IP 계층 구조와 프로토콜

TCP/IP 모델의 이름 때문에 굉장히 자주 출제되는 **최고의 함정 문제**이다. 결론부터 말하면 **"인터넷 계층"에는 TCP 프로토콜이 없다!**
(TCP/IP라는 통신 규약 이름은 그저 이 구조에서 가장 많이 일하는 핵심 두 녀석, 즉 **전송 계층의 TCP**와 **인터넷 계층의 IP**의 이름을 합쳐서 부르는 것뿐이다.)

| TCP/IP 4계층                   | 대응되는 OSI 계층              | 주요 소속 프로토콜 (시험 단골) | 핵심 역할                                                             |
| ------------------------------ | ------------------------------ | ------------------------------ | --------------------------------------------------------------------- |
| **4. 응용 계층 (Application)** | 5, 6, 7계층 (응용, 표현, 세션) | HTTP, FTP, SSH, DNS, SMTP      | 웹 브라우저, 메일 등 사용자 프로그램이 직접 통신하는 계층             |
| **3. 전송 계층 (Transport)**   | 4계층 (전송)                   | **TCP**, UDP                   | 데이터를 어떤 방식(안전 꼼꼼 TCP vs 빠르고 대충 UDP)으로 보낼지 결정  |
| **2. 인터넷 계층 (Internet)**  | 3계층 (네트워크)               | **IP**, ICMP, ARP, RARP        | 패킷에 주소(IP)를 붙여서 목적지까지 최적의 길(라우팅)을 찾아주는 역할 |
| **1. 네트워크 접속 계층**      | 1, 2계층 (물리, 데이터링크)    | Ethernet(이더넷), MAC, X.25    | 실제 랜선이나 무선(물리적 기기)으로 0과 1의 전기 신호가 오가는 계층   |

**(💡 주의 포인트):**
시험 문제에서 _"인터넷 계층에 속하는 프로토콜을 고르시오"_ 라고 질문했는데, 보기에 그럴싸하게 **TCP**가 있다면 무조건 걸러야 하는 오답이다. TCP는 **전송 계층(3계층)** 소속이다.

### ICMP (Internet Control Message Protocol)

인터넷 계층(네트워크 파트) 소속 프로토콜로, IP 프로토콜을 도와 통신 중 발생하는 **'오류나 상태를 확인(Control Message)'**해 주는 녀석이다.

- **네트워크 진단**: 목적지에 도달하지 못했거나(오류), 회선이 혼잡할 때 원격지 호스트에게 "야, 너한테 가는 길 끊겼어!" 혹은 "천천히 좀 보내!" 하고 제어 메시지를 전달한다.
- 대표적인 명령어: **`ping`**. 우리가 인터넷이 되는지 확인할 때 치는 `ping` 명령어가 바로 이 ICMP 프로토콜의 _에코 요청(Echo Request)_ 메시지를 활용한 대표적인 친구다.

### FTP (File Transfer Protocol)

네트워크 상에서 **파일을 주고받기(전송) 위해 사용하는 가장 대표적인 응용 계층 프로토콜**이다.

- **2개의 포트 번호 사용**: 매우 독특하게 작업 지시용과 실제 데이터 전송용, 두 개의 문(포트)을 쓴다.
  - **포트 21번**: 제어 포트 (연결, 로그인, 명령어 전달)
  - **포트 20번**: 데이터 포트 (실제 파일이 오가는 통로, Active 모드 시)
- **보안 취약점**: 이름 그대로 순수하게 '파일 전송'만 생각해서 만든 옛날 규약이라 비밀번호나 파일을 암호화하지 않고 생짜(평문)로 보낸다. 그래서 요즈음은 보안을 입힌 **SFTP (SSH 기반)** 방식 등을 권장한다.

### OSI 상위 3계층 (응용, 표현, 세션 계층)

가장 사용자와 가깝게 맞닿아 상호작용하는 소프트웨어(응용 프로그램) 중심의 계층들이다. (TCP/IP 4계층 모델에서는 이 세 개를 통째로 묶어 '응용 계층'이라고 하나로 부른다.)

- **5계층. 세션 계층 (Session Layer)**
  - **키워드**: 연결 유지, 동기화(체크포인트)
  - 두 컴퓨터 간의 대화(세션) 흐름을 관리한다. 전송 중 연결이 끊겼을 때 처음부터 다시 보내지 않도록 중간중간 **'점검점(동기점)'**을 찍어 관리한다.
- **6계층. 표현 계층 (Presentation Layer)**
  - **키워드**: 암호화/해독, 압축, 데이터 양식 변환(번역)
  - 송수신자가 각기 다른 형식(포맷)을 쓰더라도 서로 알아볼 수 있게 문맥(JPEG, ASCII 등)을 일치시켜 주는 번역가다. 데이터(파일)의 **암호화와 압축**이 주로 이곳에서 이루어진다.
- **7계층. 응용 계층 (Application Layer)**
  - **키워드**: 사용자/프로그램 인터페이스
  - 웹 브라우저, 카카오톡, 이메일처럼 사용자가 **직접 눈으로 보고 조작**하며 네트워크 서비스(HTTP, FTP, SMTP 등)를 이용하는 제일 최상단 출입문이다.

### 주요 프로토콜과 포트 번호 조합

시스템 및 네트워크 관련 시험에서 '무조건 1문제 이상 출제되는' 필수 암기 테이블이다. 포트 번호(문)를 알아야 방화벽도 열고 데몬(서비스)도 세팅할 수 있다.

| 프로토콜   | 포트 번호  | 용도 및 특징                                                                                               |
| ---------- | ---------- | ---------------------------------------------------------------------------------------------------------- |
| **FTP**    | `20`, `21` | 파일 전송 (20: 데이터, 21: 제어)                                                                           |
| **SSH**    | `22`       | **원격 접속의 표준**. 패킷을 암호화하여 매우 안전함. (과거 평문으로 보내던 Telnet(`23`)을 완벽하게 대체함) |
| **Telnet** | `23`       | 원격 접속 (하지만 보안이 취약하여 현재는 거의 쓰지 않고 시험에 오답 단골로 등장)                           |
| **SMTP**   | `25`       | **이메일 송신(발송)** 프로토콜. (보낼 때 쓴다!)                                                            |
| **DNS**    | `53`       | 도메인 이름(www.google.com)을 IP 번호로 찰떡같이 바꿔주는 전화번호부                                       |
| **HTTP**   | `80`       | 웹 브라우저(통신) 기본 규약                                                                                |
| **POP3**   | `110`      | **이메일 수신** 프로토콜. (받을 때 쓴다! 이메일을 서버에서 내 PC로 가져옴)                                 |
| **IMAP**   | `143`      | **이메일 수신** 프로토콜. (POP3와 달리 메일이 서버에도 그대로 남아있어 여러 기기 연동에 유리)              |
| **HTTPS**  | `443`      | HTTP에 보안(SSL/TLS 암호화)을 추가한 웹 통신 규약                                                          |

### IPv4 주소 클래스 (A, B, C, D, E 클래스)

전체 IP 주소를 국가나 기관, 일반 가정 등 네트워크 '규모(크기)'에 따라 효율적으로 배분하기 위해 5개의 등급(Class)으로 나눈 구조이다.

| 클래스       | 시작 비트 (이진수) | 첫 번째 자리 범위 | 배정 대상 및 규모                     | 주요 특징                                                              |
| ------------ | ------------------ | ----------------- | ------------------------------------- | ---------------------------------------------------------------------- |
| **A 클래스** | `0`                | **0 ~ 127**       | 국가망, 대형 통신망                   | 호스트(PC)를 무려 **1,677만 개** 넘게 쓸 수 있는 제일 큰 블록          |
| **B 클래스** | `10`               | **128 ~ 191**     | 중대형 네트워크 (기업, 학교 등)       | 호스트 약 6만 5천 개 사용 가능                                         |
| **C 클래스** | `110`              | **192 ~ 223**     | 소규모 네트워크 (가정용 공유기, PC방) | 호스트 **254개** 사용 가능. 우리가 흔히 쓰는 `192.168.x.x`가 여기 속함 |
| **D 클래스** | `1110`             | **224 ~ 239**     | 멀티캐스트(Multicast) 전용            | 여러 명에게 동시에 똑같은 방송을 쏠 때(IPTV 등) 쓰는 특수 목적용       |
| **E 클래스** | `1111`             | **240 ~ 255**     | 실험 및 연구용                        | 일반 시장에는 풀지 않고 미래를 위해 남겨둔 예비(실험용) 주소           |

**(💡 꿀팁 포인트):**
시험에서는 보통 **"192.168.0.1은 무슨 클래스인가?"(정답: C)** 나 **"멀티캐스트용으로 쓰이는 클래스는?"(정답: D)** 같은 첫 번째 자리 범위 숫자나 특수 목적 클래스를 주로 묻는다.

### 루프백 (Loopback) 주소

자기 자신을 가리키는 **가상의 IP 주소**이자 논리적인 네트워크 인터페이스이다. 즉, 출발지와 목적지가 똑같이 **자기 자신(내 PC)**인 주소이다.

- **루프백 IP 주소**: **`127.0.0.1`** (IPv4 기준)
  - `127.x.x.x` 대역은 오직 이 루프백 용도로만 예약되어 있어, 인터넷에서 실제 다른 컴에게 배정될 수 없다.
  - IPv6 기준으로는 `::1` 이라고 쓴다.

* **주요 용도 (네트워크 점검)**: 이 IP 주소(`127.0.0.1`)로 핑(ping)을 때려보거나 데이터를 보냈을 때 무사히 돌아오면, "아, 내 컴퓨터의 랜카드가 고장나지 않았고 네트워크(TCP/IP 프로토콜) 기본 설정도 정상적으로 돌아가는구나!" 하고 **자기 자신의 네트워크 상태를 점검**할 때 쓴다.
* 웹 개발을 할 일반 컴퓨터(로컬)에서 띄운 서버"에 접속할 때 쓰는 그 친숙한 **`localhost`**가, 컴퓨터가 알아듣는 IP 숫자로 바뀌면 바로 이 `127.0.0.1`이 된다.
* **`lo` vs `lo0`의 차이 (인터페이스 이름)**
  - 역할은 100% 동일한 **"루프백 장치(인터페이스)의 이름"**일 뿐이다. 운영체제(OS)의 계보나 환경에 따라 이름표만 다르게 부른다.
  - **`lo`**: 리눅스(Linux) 계열에서 부르는 기본 루프백 인터페이스 이름 (예: `ifconfig` 입력 시 `lo`가 뜬다.)
  - **`lo0`**: 유닉스(Unix), BSD 계열 또는 Mac OS 등에서 부르는 루프백 인터페이스 이름 (리눅스 시스템 관리 문제에서 오답 보기에 가끔씩 섞여 나온다.)

### 주요 최상위 도메인 (TLD) - org, net 등

인터넷 주소(도메인명) 맨 끝부분에 붙는 **최상위 도메인(Top-Level Domain, TLD)**은 해당 사이트를 운영하는 **기관의 성격이나 목적**을 나타낸다. (영문 약자를 보면 쉽게 유추할 수 있다.)

- **`.com` (Commercial)**: 영리를 목적으로 하는 **일반 기업(상업 기관)**. (구글, 네이버 등 가장 대중적)
- **`.net` (Network)**: 인터넷 관련 망을 관리하거나 서비스를 제공하는 **네트워크 관련 기관**용. (하지만 현재는 `.com`처럼 일반적인 목적으로도 구별 없이 광범위하게 쓰인다.)
- **`.org` (Organization)**: 이익을 쫓지 않는(비영리) **일반 기관 및 단체**용 도메인. (예: 위키백과(wikipedia.org), 시민단체 등)
- **`.edu` (Education)**: 미국의 **교육 기관(대학 등)** 전용. (한국은 `.ac.kr`이나 `.hs.kr` 형태를 주로 쓴다.)
- **`.gov` (Government)**: 미국의 **정부 기관** 전용. (한국은 `.go.kr`을 쓴다.)
- **`.mil` (Military)**: 미국의 **군사 기관** 전용.

### 넷마스크(Netmask)와 기본 게이트웨이(Default Gateway)

인터넷(네트워크)을 설정할 때 반드시 세트로 입력해야 하는 핵심 3요소(IP 주소, 넷마스크, 기본 게이트웨이) 중 2가지이다.

- **넷마스크 (서브넷 마스크, Netmask/Subnet Mask)**
  - **역할**: "이 IP 중에서 어디까지가 우리 동네(네트워크) 주소이고, 어디부터가 내 집(호스트) 주소인지"를 쓱 덮어서(Mask) 명확히 **가려주는 필터** 역할을 한다.
  - 똑같은 동네(같은 네트워크 대역)에 있는 PC인지 아닌지 판별할 때 쓴다. (예: `255.255.255.0` => 앞의 3칸까지 일치해야 같은 동네 묶음이다!)
- **기본 게이트웨이 (Default Gateway / Gateway Address)**
  - **역할**: 우리 동네(내부망)에서 다른 동네(외부 인터넷망 등)로 나갈 때 무조건 통과해야 하는 **마을의 유일한 '출구(수문장, 라우터 IP)'**를 의미한다.
  - 넷마스크로 계산해 보니 상대방이 "우리 동네 애가 아니네?" 싶으면, 무조건 이 '기본 게이트웨이' 주소로 데이터를 던져서 바깥(인터넷)으로 내보낸다.

### ARP (Address Resolution Protocol)

네트워크에서 통신할 때 **"논리적인 IP 주소"를 "물리적인 MAC 주소(랜카드 고유번호)"로 바꾸어 주는 해결사(Resolution) 프로토콜**이다.

- **왜 필요할까?**: 인터넷 세상에서 길을 찾을 때는 논리적인 'IP 주소'를 쓰지만, 막상 스위치나 허브를 거쳐 내 PC 랜카드에 선이 꽂히는 진짜 **마지막 물리적 전달 단계(2계층)**에서는 그 고유한 하드웨어 번호인 **'MAC 주소'**가 반드시 필요하기 때문이다.
- **작동 방식 (동네 방송)**: 송신 측이 "혹시 192.168.0.5를 쓰는 사람 있니? 네 실제 집 주소(MAC 주소) 좀 알려줘!" 하고 전체에게 크게 무전(Broadcasting)을 치면, 그 번호를 가진 진짜 주인이 "어! 나야! 내 MAC 주소는 이거(`AA:BB:CC:...`)야!" 하고 알려주는 방식이다.
- **↔️ RARP (Reverse ARP)**: 반대로 **MAC 주소를 IP 주소로 바꿔주는** 반대 과정의 프로토콜이다. (시험 보기에서 짝꿍으로 자주 등장함)

### mii-tool (Media Independent Interface Tool)

네트워크 인터페이스(랜카드)의 **물리적 연결 상태(Link Status)를 확인하거나 연결 속도/방식을 강제로 설정**할 때 사용하는 리눅스 구형 명령어이다.

- **네트워크 물리적 점검맨**: 선이 제대로 꽂혀 있는지("link ok"), 현재 속도가 100Mbps인지 1000Mbps인지(기가비트), 양방향 통신(Full Duplex) 중인지 반이중 통신(Half Duplex) 중인지를 바로 알려준다.
- 요즘 등장하는 최신 리눅스 환경과 최신 1급 시험에서는, 이 구형 `mii-tool` 명령어 대신 훨씬 최신이고 많은 기능을 제공하는 **`ethtool`**이 정답으로 출제되는 빈도가 높다. (두 명령어의 역할은 서로 겹친다.)

### 서브네팅 (넷마스크 숫자만 보고 10초 만에 풀기)

이진수 비트 계산이 헷갈린다면, **256에서 넷마스크 끝자리를 빼는 공식** 하나만 기억하면 단순 산수 문제로 바뀐다.

- **핵심 마법의 숫자: `256`**
  - 하나의 기본 C클래스는 총 256개의 IP 주소(0~255)를 가진다.
- **방 하나당 들어가는 전체 IP 개수 구하기 (단위 구하기)**
  - **공식**: `256 - (주어진 넷마스크의 마지막 숫자)`
  - 예) 넷마스크가 `255.255.255.192`로 주어졌다면? $\to 256 - 192 = \mathbf{64}$개. (한 묶음에 64개의 IP가 들어간다는 뜻)
- **PC에 꽂을 수 있는 진짜 IP(실제 호스트) 개수 구하기**
  - **공식**: 위에서 구한 숫자에서 무조건 **`- 2`**를 뺀다.
  - 예) 아까 64가 나왔으므로 $\to 64 - 2 = \mathbf{62}$대. (실제 PC 62대에만 IP 부여 가능)
  - **왜 `- 2`를 뺄까? (시험 단골 함정)**: 제일 처음 시작하는 번호(네트워크 이름 용도)와 제일 마지막 번호(전체에게 외치는 브로드캐스트 용도) 2개는 규칙상 컴퓨터에 절대로 못 주기 때문이다.
- **서브네트워크 개수 구하기 (총 몇 개의 묶음으로 쪼개졌는가?)**
  - **공식**: `256 / (맨 처음에 구한 묶음 숫자)`
  - 예) 전체 256개를 한 방망이(64개)로 몇 번 쪼갤 수 있나? $\to 256 \div 64 = \mathbf{4}$묶음(네트워크가 4개로 쪼개짐).

### ss (Socket Statistics) 명령어

네트워크 소켓(포트)의 상태와 트래픽 연결 정보를 확인하는 명령어로, 기존의 무겁고 느린 **`netstat` 명령어를 완전히 대체**하는 최신 도구이다.

- **왜 `netstat` 대신 `ss`를 쓸까?**: `ss` 명령어가 커널에서 직접 정보를 가져와서 속도가 훨씬 빠르고(성능 최적화), 보여주는 정보의 양도 압도적으로 많기 때문이다. 시스템에 열린 포트나 연결된 클라이언트를 확인할 때 가장 많이 쓴다.
- **시험 단골 옵션 세트 (`-antp`)**
  - 많이 쓰이는 옵션들을 합치면 보통 `ss -antp`처럼 쓴다.
  - `-a` (all): 활성화된 것, 대기 중(LISTENING)인 것 가리지 않고 **모든** 소켓(포트) 보여줘.
  - `-n` (numeric): 포트 번호를 이름(http) 말고 그냥 **숫자(80)**로 직관적으로 보여줘. (도메인을 IP로 찾는 과정이 빠져서 속도 향상)
  - `-t` (tcp): 오직 **TCP** 소켓만 모아서 보여줘. (`-u`는 UDP)
  - `-p` (process): 지금 이 문(포트)을 열고 있는 **프로세스(프로그램) 이름과 PID번호**도 같이 보여줘.

### 서브네팅 기출문제 풀이 ("넷마스크가 255.255.255.192일 때 할당 가능한 IP 주소는?")

1. **방 크기(블록 단위) 구하기**: `256 - 192 = 64`. (즉, 전체 IP 주소가 `64`개씩 한 묶음으로 쪼개진다.)
2. **쪼개진 4개의 네트워크 방(대역) 범위 나열하기**:
   - **1번 방**: `0 ~ 63`
   - **2번 방**: `64 ~ 127`
   - **3번 방**: `128 ~ 191`
   - **4번 방**: `192 ~ 255`
3. **일반 컴퓨터에 "할당 가능한" 진짜 IP 범위 구하기 (`-2` 법칙)**:
   각 방에서 **맨 첫 번호(네트워크 주소)**와 **맨 끝 번호(브로드캐스트 주소)**를 빼면, 그것이 컴퓨터에 넣을 수 있는 IP(유효 호스트)다.
   - 1번 방 할당 가능 IP: ~~`0`~~ / **`1 ~ 62`** / ~~`63`~~
   - 2번 방 할당 가능 IP: ~~`64`~~ / **`65 ~ 126`** / ~~`127`~~
   - 3번 방 할당 가능 IP: ~~`128`~~ / **`129 ~ 190`** / ~~`191`~~
   - 4번 방 할당 가능 IP: ~~`192`~~ / **`193 ~ 254`** / ~~`255`~~

### 서브네팅 실전 계산문제 ("IP가 192.168.3.68 이고 넷마스크가 255.255.255.192일 때, 네트워크 주소와 브로드캐스트 주소는?")

특정 IP가 주어지고 이들이 속한 네트워크의 **시작 번호(네트워크 주소)**와 **끝 번호(브로드캐스트 주소)**를 찾는 문제도 10초 컷으로 똑같이 풀면 된다.

1. **방 크기(블록 단위) 구하기 똑같이 적용!**
   - `256 - 192 = 64`. (마지막 자리가 `64`개씩 쪼개진다.)
   - 즉, 끝자리 범위는 (0~63), (64~127), (128~191), (192~255)가 된다.
2. **이 IP는 몇 번 방에 들어갈까? (내 집 찾기)**
   - 제시된 IP 주소의 제일 끝자리 **`68`**을 방금 구한 방 범위들에 넣어본다.
   - `68`은 **2번 방 범위(`64 ~ 127`)** 안에 쏙 들어간다!
3. **네트워크 주소와 브로드캐스트 주소 건지기 (정답 구하기)**
   해당 방(2번 방) 전체 범위의 **제일 첫 번호와 마지막 번호**가 그대로 정답이 된다. 앞의 `192.168.3.` 부분은 문제없이 똑같이 붙여준다.
   - **네트워크 주소 (맨 첫 번호)**: `192.168.3.` + `64` $\to$ **`192.168.3.64`**
   - **브로드캐스트 주소 (맨 끝 번호)**: `192.168.3.` + `127` $\to$ **`192.168.3.127`**

(💡 **시험 꿀팁**: 이 문제에서 가장 많이 하는 실수는 넷마스크에 쓰여있던 `192`라는 숫자를 정답에 섞어버리는 것입니다. 넷마스크는 오직 '64개씩 쪼개라'는 지시사항(결정타)일 뿐, 결과 IP값에는 절대 들어가지 않습니다!)
